### Răspuns brutal de sincer: **NU, o diagramă de secvențe NU este de ajuns ca să înțelegi tot flow-ul aplicației.**

Iată de ce, ca viitor Software Engineer, trebuie să folosești ambele instrumente (și nu doar unul):

#### 1. Ce arată excelent Diagrama de Secvențe?
* **Ordinea temporală**: Îți arată exact ordinea pașilor (Pasul 1, 2, 3 în timp).
* **Relația directă de apel**: Vezi cine pe cine apelează (Client -> Router -> Storage -> Memorie).
* **Ramificarea simplă**: Poți vedea un caz fericit vs. un caz de eroare (de ex. blocul `alt` de utilizator inexistent / existent).

#### 2. Ce OMITÉ Diagrama de Secvențe? (De ce ai nevoie și de Diagrama de Flow/Arhitectură)
* **Topologia statică (Structura de fișiere)**: Diagrama de secvențe nu îți spune unde se află codul fizic pe disc, cum sunt împărțite modulele sau ce importuri există (de ex. că `Storage` este importat în `main.py`).

## 🧐 Contra-Atacul Arhitectural: Ce am greșit în fix-urile simple?

### 1. Conconcurență & Condiții de cursă (TOCTOU) în Bug-ul #1
* **Problema din cod**: În `main.py`, mai întâi verificăm dacă utilizatorul există (`storage.get_user`), iar apoi creăm evenimentul (`storage.create_event`).
* **Critica**: Într-un mediu cu mai multe fire de execuție (cum rulează FastAPI în threadpool), între cele două apeluri utilizatorul poate fi șters de un alt thread. Acest lucru se numește **TOCTOU (Time-of-Check to Time-of-Use)** și generează „evenimente orfane” în memorie.
* **Soluția Corectă**: Validarea integrității relaționale (asemănător unei chei externe - Foreign Key în SQL) ar trebui să fie responsabilitatea exclusivă a bazei de date (sau a clasei `Storage`), protejată de un lock.

### 2. Performanță dezastruoasă la Paginare ($O(N)$ timp și spațiu) în Bug-ul #2
* **Problema din cod**: `all_events = [e for e in self._events.values() if e.deleted_at is None]` urmat de slice.
* **Critica**: Dacă avem 1.000.000 de evenimente în memorie, la fiecare request de paginare (chiar și pentru doar 5 elemente), Python parcurge toate cele un milion de înregistrări, creează o listă temporară masivă în RAM, o copiază și apoi face slice. Acest lucru va bloca procesorul și va genera presiune uriașă pe Garbage Collector (GC).
* **Soluția Corectă**: 
  * Menținerea unui **Index Activ** separat (de ex. `self._active_events` ca un dicționar separat din care ștergem fizic cheile când facem soft-delete).
  * Folosirea `itertools.islice` (un generator leneș) pentru a prelua doar bucata dorită fără a aloca liste intermediare mari în memorie.

### 3. Mutabilitate partajată ("Spooky Action at a Distance") în Bug-ul #3
* **Problema din cod**: `event.deleted_at = datetime.now(...)`
* **Critica**: Modelele Pydantic sunt mutabile implicit în Python. Când facem `storage.get_event` sau `list_events`, API-ul primește **o referință directă** la obiectul din memoria bazei de date. Dacă API-ul sau un alt proces modifică acea referință, starea bazei noastre de date se corupe în fundal, fără control!
* **Soluția Corectă**: Pydantic models ar trebui setate ca **imobile (frozen/immutable)**. La orice modificare, storage-ul ar trebui să creeze o copie nouă folosind `.model_copy(update=...)`, returnând doar copii către exterior.

## 🛠️ Alternativa Profesionistă (Refactorizarea Corectă)

Dacă am dori o soluție pregătită pentru producție (thread-safe, imutabilă și eficientă), codul ar fi trebuit să arate așa:

### A. În `app/models.py` (Modele imobile / Frozen):
```python
class Event(BaseModel):
    model_config = ConfigDict(frozen=True) # Utilizatorul nu poate modifica obiectele direct
    id: int
    user_id: int
    event_type: EventType
    metadata: dict = Field(default_factory=dict)
    created_at: datetime = Field(default_factory=_now)
    deleted_at: Optional[datetime] = None
```

### B. În `app/storage.py` (Adăugarea lock-urilor și a Indexului Activ):
```python
import threading
import itertools

class Storage:
    def __init__(self) -> None:
        self._users: dict[int, User] = {}
        self._events: dict[int, Event] = {}
        self._active_events: dict[int, Event] = {} # INDEX ACTIV pentru O(1) delete și listare rapidă
        self._lock = threading.Lock() # Lock pentru Thread-Safety
        self._next_user_id = 1
        self._next_event_id = 1

    def create_event(self, data: EventCreate) -> Event:
        with self._lock:
            # Enforțăm integritatea relațională la nivel de bază de date
            if data.user_id not in self._users:
                raise UserNotFoundError("User not found")
            
            event = Event(
                id=self._next_event_id,
                user_id=data.user_id,
                event_type=data.event_type,
                metadata=data.metadata,
            )
            self._events[event.id] = event
            self._active_events[event.id] = event
            self._next_event_id += 1
            return event.model_copy() # Returnăm o copie!

    def list_events(self, offset: int = 0, limit: int = 50) -> list[Event]:
        with self._lock:
            # O(offset + limit) timp, O(limit) spațiu folosind generator leneș
            iterator = self._active_events.values()
            sliced = list(itertools.islice(iterator, offset, offset + limit))
            return [e.model_copy() for e in sliced] # Întoarcem doar copii

    def soft_delete_event(self, event_id: int) -> Optional[Event]:
        with self._lock:
            event = self._events.get(event_id)
            if event is None or event.deleted_at is not None:
                return None
            
            # Creăm un obiect nou în loc să îl modificăm pe cel vechi in-place (Imutabilitate)
            updated = event.model_copy(update={"deleted_at": datetime.now(timezone.utc)})
            self._events[event_id] = updated
            self._active_events.pop(event_id, None) # Eliminare O(1) din indexul activ
            return updated
```

## 🔍 Ce a identificat subagentul?

### 1. Bug Logic la Fusul Orar (Timezones) — *CRITICAL*
* **Problema din planul inițial**: Am propus să facem blind `.replace(tzinfo=timezone.utc)` pe parametrul `since`.
* **De ce e grav**: Dacă clientul trimite o dată care are deja fus orar (de ex: `+03:00`), `.replace` nu convertește timpul, ci doar rescrie eticheta de fus orar cu `UTC`. Asta mută ora cu 3 ore în viitor/trecut (decalaj de dată).
* **Soluția Corectă**:
  ```python
  if since.tzinfo is None:
      since = since.replace(tzinfo=timezone.utc) # Adăugăm UTC dacă e naive
  else:
      since = since.astimezone(timezone.utc) # Convertim corect la UTC dacă are deja alt fus orar
  ```

### 2. Scurgerea Responsabilităților (Architecture Smell)
* **Problema**: Normalizarea datelor de tip query (fus orar) nu ar trebui să se facă în baza de date (`storage.py`), ci în controller-ul API (`main.py`). Baza de date trebuie să primească doar date curate și standardizate.
* **Soluția**: Vom face curățarea datei `since` în `app/main.py` înainte de a apela storage-ul.

### 3. Lipsa paginării & Performanță $O(N)$
* **Critica**: Fără paginare, dacă un utilizator are 50.000 de evenimente, API-ul le va returna pe toate deodată, putând bloca serverul. De asemenea, scanarea liniară a tuturor evenimentelor din memorie pentru a le filtra după `user_id` este ineficientă la dataset-uri mari.
* **Compromisul (Tradeoff-ul de Simplitate)**: 
  Conform regulii noastre **"Simplicity First"** și instrucțiunilor stricte din [README.md](README.md), paginarea pe acest endpoint **nu a fost solicitată**. Astfel, pentru a nu complica codul inutil pentru un task de internship și pentru a păstra totul simplu:

## 🏗️ 1. Analiza SOLID (Arhitectură)

### **S** – Single Responsibility Principle (SRP)
* 🔴 **Problema**: Clasa `Storage` încalcă SRP. Ea se ocupă în același timp de:
  1. Inițializarea bazei de date in-memory (`__init__` și `reset`).
  2. Gestiunea utilizatorilor (creare, lookup).
  3. Gestiunea evenimentelor (creare, paginare, soft-delete, filtrare).
* 🟢 **Soluția SOLID**: Divizarea în două depozite distincte: `UserRepository` și `EventRepository`.
* 🔴 **Problema în API**: Endpoint-ul `GET /users/{user_id}/events` se ocupă de rutare HTTP, validare de existență, dar și de normalizarea datelor naive de tip datetime (care e o regulă de transformare a inputului).
* 🟢 **Soluția SOLID**: Normalizarea fusului orar ar trebui făcută automat prin validarea Pydantic (folosind un validator `@model_validator` sau un tip custom) înainte de a ajunge în corpul funcției endpoint.

### **O** – Open/Closed Principle (OCP)
* 🔴 **Problema**: În prezent, routerele din `main.py` depind direct de clasa concretă `Storage`. Dacă vrem să trecem de la in-memory la o bază de date reală (PostgreSQL cu SQLAlchemy), trebuie să rescriem codul în `main.py` și `storage.py`. Codul nu este deschis pentru extindere.
* 🟢 **Soluția SOLID**: Definirea unei interfețe abstracte (de ex. `AbstractEventRepository` folosind `abc.ABC`). Implementăm apoi `InMemoryEventRepository` și `PostgreSQLEventRepository`. Routerele vor consuma interfața abstractă, fără să le pese de implementarea din spate.

### **L** – Liskov Substitution Principle (LSP)
* 🟢 **Status**: Nu avem o ierarhie de moștenire, deci nu este încălcat. Însă dacă s-ar crea clase derivate din `Storage`, ele ar trebui să poată fi substituite fără a modifica comportamentul API-ului (de ex. ridicarea acelorași excepții de domeniu).

### **I** – Interface Segregation Principle (ISP)
* 🔴 **Problema**: O clasă controller care are nevoie doar de utilizatori (de ex. `POST /users`) este forțată să depindă de o clasă `Storage` care conține și metode de evenimente.
* 🟢 **Soluția SOLID**: Separarea metodelor de User de cele de Event în interfețe separate (`IUserStorage`, `IEventStorage`).

### **D** – Dependency Inversion Principle (DIP)
* 🔴 **Problema**: `app/main.py` importă o instanță globală concretă: `from app.storage import storage`. Aceasta cuplează strâns stratul API de implementarea bazei de date.
* 🟢 **Soluția SOLID**: Folosirea sistemului de **Dependency Injection** din FastAPI (`Depends`).
  ```python
  @app.get("/users/{user_id}/events")
  def get_user_events(user_id: int, db: AbstractStorage = Depends(get_db)):
      ...
  ```

---

## ⚡ 2. Analiza de Performanță

### A. Complexitatea Timp (Time Complexity)
1. **Verificarea existenței utilizatorului**: `storage.get_user(user_id)` este **$O(1)$** (căutare directă după cheie în dicționarul de utilizatori). Excelent.
2. **Preluarea evenimentelor utilizatorului (`get_user_events`)**:
   * 🔴 **Abordarea Naivă (Planificată)**: **$O(N)$**, unde $N$ este **numărul total de evenimente din sistem**. Scanăm liniar `self._events.values()` pentru a găsi evenimentele unui singur utilizator. Cu 1.000.000 de evenimente totale în memorie, interogarea va fi extrem de lentă chiar dacă utilizatorul respectiv are un singur eveniment.
   * 🟢 **Abordarea Optimizată**: **$O(M)$**, unde $M$ este **numărul de evenimente ale utilizatorului respectiv** ($M \ll N$). Putem face asta menținând un index secundar în `Storage`: `self._user_events: dict[int, list[Event]]`.

### B. Complexitatea Spațiu (Space Complexity)
* Pentru noul endpoint, stocăm în memorie doar lista filtrată de evenimente returnate. Spațiul este de **$O(M)$** (unde $M$ este numărul de evenimente returnate). Este acceptabil, dar fără un parametru `limit` pe acest endpoint, memoria consumată la serializare poate crește necontrolat.

### C. Presiunea pe Garbage Collector (GC)
* Din cauza list-comprehensions din storage (`[e for e in ...]`), la fiecare request API-ul generează liste intermediare mari în memorie care sunt imediat aruncate după trimiterea răspunsului JSON. Acest comportament crește frecvența declanșării GC-ului în Python, generând micro-pauze (jitter) în latența API-ului.

## 🔍 Ce am găsit în analize (Bune Practici & Instrumente)

### 1. **Metoda Commits-urilor Semantice (Conventional Commits)**
Este cel mai popular standard în industrie. Definește o structură clară pentru titlul commit-ului:
``<type>(<scope>): <descriere scurtă la imperativ>``
* **Tipuri (`type`) comune**:
  * `feat`: O funcționalitate nouă.
  * `fix`: Rezolvarea unui bug.
  * `docs`: Modificări doar în documentație.
  * `style`: Formatare, spații, denumiri (fără modificări de logică).
  * `refactor`: Restructurarea codului existent.
  * `test`: Adăugarea sau corectarea testelor.
  * `chore`: Dependințe, build-uri, fișiere gitignore, etc.
* **Imperativ**: Subiectul trebuie scris la modul imperativ (ex. „*Add validation*” în loc de „*Added validation*” sau „*Adds validation*”).

### 2. **Metoda Commitizen (Interactivă)**
Este o unealtă CLI (bazată pe Node.js sau Python) care înlocuiește comanda standard `git commit` cu un vrăjitor (wizard) interactiv rulează prin `git cz` sau `cz commit`. Acesta te întreabă pas cu pas tipul, scopul, descrierea scurtă, corpul și dacă există breaking changes sau tickete închise.

### 3. **Metoda Template-urilor Git (Git Commit Templates)**
O variantă extrem de elegantă și nativă (nu necesită instalarea de dependințe terțe). Putem defini un șablon local într-un fișier (ex. `.gitmessage`) și să configurăm Git să îl folosească. Când rulezi `git commit` (fără `-m`), Git deschide editorul tău favorit pre-populat cu structura corectă și instrucțiuni comentate.

### 4. **Metoda Hook-urilor Git & Lintere de Commits**
Folosirea unor utilitare precum `husky` și `commitlint` pentru a rula scripturi de verificare (linter) înainte ca un commit să fie creat. Dacă mesajul scris nu respectă standardul Conventional Commits, commit-ul este respins automat.

---

## 2. Critici constructive și Îmbunătățiri Aduse

Subagentul a evidențiat câteva puncte slabe arhitecturale și de optimizare, pe care le-am rezolvat sau documentat:

### ⚡ A. Filtrare ineficientă cu dublă trecere (Rezolvat)
* **Critica**: În varianta anterioară, [`Storage.get_user_events`](app/storage.py#L63-L71) făcea două iterări pe listă și aloca de două ori memorie când `since` era prezent (o dată pentru filtrarea soft-delete, a doua oară pentru `since`).
* **Acțiunea noastră**: Am refactorizat codul din [`app/storage.py`](app/storage.py) pentru a folosi o **singură listă de comprehensiune** (o singură trecere $O(M)$):
  ```python
  return [
      e
      for e in user_events
      if e.deleted_at is None and (since is None or e.created_at > since)
  ]
  ```
  Modificarea a fost testată (`pytest` trece integral), staged și push-uită în commit-ul `70e0e6c`.

### 🧵 B. Siguranța firelor de execuție (Thread-safety)
* **Critica**: Deoarece FastAPI rulează request-uri în mod concurent, dicționarele noastre in-memory nu sunt protejate împotriva accesului concurent. O scriere simultană cu o iterare poate declanșa `RuntimeError` din partea interpretorului Python.
* **Status**: Am documentat această limitare în [`NOTES.md`](NOTES.md#L51) în secțiunea *Ce-ai face cu mai mult timp* (propunând un `threading.Lock`). Ea este acceptabilă în faza de prototip in-memory, dar critică în producție.

### 🛑 C. Lipsa paginării (Limită de performanță / DoS)
* **Critica**: Endpoint-ul `/users/{user_id}/events` nu are parametri `offset` sau `limit` (spre deosebire de `/events`). Dacă un utilizator strânge milioane de evenimente, apelul API le va serializa pe toate o dată, riscând un atac DoS sau epuizarea memoriei RAM.
* **Status**: Am documentat acest compromis în note ca decizie de design bazată pe regula **Simplicity First** (nu a fost cerut explicit în cerințe). În producție, am adăuga obligatoriu paginare.

### 🌍 D. URL-Encoding la Parametrii de Timezone
* **Critica**: Transmiterea unui fus orar local de tipul `+03:00` va genera o eroare `422` dacă caracterul `+` nu este URL-encoded în `%2B` (deoarece serverele web decodifică `+` ca spațiu).
* **Status**: Am adăugat acoperire exactă de test în `test_get_user_events_with_since` (care folosește `%2B03:00`) pentru a documenta și garanta acest comportament corect.

## 🧪 Testele adăugate în „Monkey Mode”

Am adăugat următoarele scenarii în [`tests/test_app.py`](tests/test_app.py#L282-L373):
* **Float inputs**: `GET /users/1.5/events` -> Returnează `422 Unprocessable Entity` (FastAPI respinge transformarea float în int).
* **Negative user IDs**: `GET /users/-5/events` -> Returnează `404 Not Found`.
* **Extremely large inputs**: Trimiterea unui ID cu zeci de cifre -> Returnează `404 Not Found` (fără overflow / server crashes).
* **Date complet malformate**: `since` cu zile invalide (`32` mai), ani non-bisecți (`29` februarie 2026), ore invalide (`25`) sau minute invalide (`60`) -> Returnează `422 Unprocessable Entity`.
* **SQL Injection strings**: Trimiterea de scripturi SQL (`1 OR 1=1`, `--;`) în path-ul `user_id` și în query-ul `since` -> Returnează `422 Unprocessable Entity` (FastAPI și Pydantic le filtrează/coersează curat).
* **Empty parameters**: Calea goală `/users//events` -> Returnează `404 Not Found`/`307 Redirect`; parametrul `since=` gol -> Returnează `200 OK` (FastAPI/Pydantic îl tratează ca `None` fără să ridice `500`).
* **Special characters**: Caractere speciale (emojis `👻` sau semne `!@#$`) în `user_id` -> Returnează `422 Unprocessable Entity`.
* **Extreme Date Boundaries**: Interogarea cu limite din viitorul depărtat (`9999-12-31`) și trecutul depărtat (`0001-01-01`) -> Returnează o listă goală, respectiv toate evenimentele active.
